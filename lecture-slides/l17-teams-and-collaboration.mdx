## Communication Overhead Grows Quadratically

In a team of *n* people, the number of communication paths is:

$$\frac{n(n-1)}{2}$$

| Team Size | Communication Paths |
|-----------|---------------------|
| 2         | 1                   |
| 4         | 6                   |
| 8         | 28                  |
| 16        | 120                 |

Communication overhead grows faster than team productivity.

Amazon has the two-pizza rule: Teams should be small enough (5-8 people) to be fed by
two large pizzas.

<aside className="notes">
- This is why Brooks' Law holds: new team members need to be brought up to speed
- Everyone spends more time coordinating and less time coding
- Important to plan upfront how to structure a team
- Transition: This is why we need communication mechanisms that scale better
</aside>

</Slide>

<Slide>

## Scalable Knowledge Sharing

**One-to-one** (doesn't scale):
- Mentorship sessions
- Hallway conversations
- Direct messages

**One-to-many** (scales better):
- Code review (synchronous, but knowledge spreads)
- Pair programming (immediate knowledge transfer)
- Q&A forums and mailing lists (asynchronous, searchable)

**One-to-many, persistent** (scales best):
- Documentation and wikis
- Recorded tech talks
- Blog posts and tutorials
- Well-commented code

<aside className="notes">
- The key is choosing communication methods that don't grow quadratically with team size
- Persistent methods are best because they can be referenced repeatedly
- Transition: So how do we apply this principle?
</aside>

</Slide>

## Write It Down


Developers see approximately a **50% productivity boost** when documentation is up-to-date, detailed, and available in multiple formats.

**Tips:**
- If you've explained something to more than two people, write it down. The third explanation should be a link to documentation.
- Fix out-of-date documentation as you work through it.

Good documentation isn't just nice to have—it's a force multiplier for the entire team.

<aside className="notes">
- Documentation reduces interruptions and scales sub-linearly with team growth
- Every documented solution is one less interruption for your teammates
- That future team member might be you in six months

</aside>
