## Unit Tests: Fast, Focused, Numerous

<p style={{fontSize: '0.95em'}}>
  Test a single "unit" — typically one class — <strong>in isolation</strong>
</p>

<ul style={{marginTop: '0.75em', fontSize: '0.9em'}}>
  <li><span style={{color: '#060'}}>✓</span> <strong>Fast:</strong> Run in milliseconds (no I/O, no network)</li>
  <li><span style={{color: '#060'}}>✓</span> <strong>Focused:</strong> When it fails, you know exactly where to look</li>
  <li><span style={{color: '#060'}}>✓</span> <strong>Numerous:</strong> Thousands of tests, run in seconds</li>
</ul>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#666'}}>
  The challenge: achieving isolation when code has dependencies
</p>

<aside class="notes">
**Unit test characteristics:**
- One test, one behavior, one class
- No databases, no files, no network
- Milliseconds to run

**The isolation problem:**
- If ThermostatController calls TemperatureSensor.readTemperature()...
- How do we test the controller without a real sensor?
- This is the central question of today's lecture

**Readability connection (L5):** Unit tests are documentation. A reader should understand the expected behavior from the test name and structure.

→ **Transition:** Integration tests cover the middle ground...
</aside>

</Slide>

## Integration Tests: Components Working Together

<p style={{fontSize: '0.95em'}}>
  Test multiple components interacting
</p>

<ul style={{marginTop: '0.75em', fontSize: '0.9em'}}>
  <li>Do components communicate correctly?</li>
  <li>Do they agree on data formats and protocols?</li>
  <li>Does timing and ordering work as expected?</li>
</ul>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#c60'}}>
  Tradeoff: slower, harder to debug when they fail
</p>

<aside class="notes">
**When integration tests shine:**
- Real ThermostatController + real TemperatureSensor + fake HVAC
- Testing the "seams" between components
- Catching issues unit tests can't: protocol mismatches, timing bugs

**When integration tests hurt:**
- Slower (database setup, network calls)
- When they fail, which component caused it?
- Harder to test edge cases

→ **Transition:** At the far end of the spectrum...
</aside>

</Slide>

<Slide>

## End-to-End Tests: The Whole System

<p style={{fontSize: '0.95em'}}>
  Test the entire system as a user would experience it
</p>

<ul style={{marginTop: '0.75em', fontSize: '0.9em'}}>
  <li><span style={{color: '#c00'}}>✗</span> <strong>Slow:</strong> Seconds or minutes per test</li>
  <li><span style={{color: '#c00'}}>✗</span> <strong>Expensive:</strong> Requires complete deployment</li>
  <li><span style={{color: '#c00'}}>✗</span> <strong>Flaky:</strong> Network glitches, timing issues, browser quirks</li>
  <li><span style={{color: '#c00'}}>✗</span> <strong>Hard to debug:</strong> Bug could be anywhere</li>
</ul>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#060'}}>
  But: tests what actually matters — does it work for users?
</p>

<aside class="notes">
**Example:** Pawtograder E2E tests:
1. Deploy complete system to test environment
2. Create class with assignments
3. Create student submissions
4. Verify student can log in, view results, submit review

**Why we still need them:**
- Catches integration issues across the whole stack
- Tests real user journeys
- But too expensive for edge case coverage

**We'll cover E2E testing in depth in L30 (GUI Patterns and Testing).**

→ **Transition:** Why not just write E2E tests for everything?
</aside>

## The Practical Mix

<p style={{fontSize: '1em'}}>
  Why not just write E2E tests for everything?
</p>

<p style={{marginTop: '1em', fontSize: '0.95em'}}>
  Imagine testing 15 edge cases for a thermostat:
</p>

<aside class="notes">
**The math doesn't work:**
- 15 edge cases × 1 minute each = 15 minutes
- Unit tests: 15 edge cases × 10ms = 150ms
- Run unit tests on every save; E2E tests nightly

**The right balance:**
- Many unit tests for edge cases and logic
- Some integration tests for component interaction
- Few E2E tests for critical user journeys

**Today's focus:** Unit testing—specifically, how to achieve isolation.

## The Dependency Problem

<p style={{fontSize: '0.95em'}}>
  Consider a <code>ThermostatController</code> in a smart home system:
</p>

```java
public class ThermostatController {
    private final TemperatureSensor sensor;
    private final HVACService hvacService;
    private final NotificationService notifier;

    public ThermostatController(TemperatureSensor sensor,
                                 HVACService hvacService,
                                 NotificationService notifier) {
        this.sensor = sensor;
        this.hvacService = hvacService;
        this.notifier = notifier;
    }
    // ...
}
```

<aside class="notes">
**Notice the dependencies:**
- TemperatureSensor — reads from hardware
- HVACService — controls actual heating/cooling
- NotificationService — sends real messages to users

**Design for change connection (L7-8):** This is dependency injection! We designed for testability.

**The question:** How do we unit test this without real hardware?

→ **Transition:** Let's see the method we want to test...
</aside>

</Slide>

<Slide>

## The Method Under Test

```java
public void adjustToTargetTemperature(double targetTemp, String zoneId) {
    double currentTemp = sensor.readTemperature(zoneId);
    double delta = targetTemp - currentTemp;

    if (Math.abs(delta) > 0.5) {
        HVACMode mode = delta > 0 ? HVACMode.HEATING : HVACMode.COOLING;
        hvacService.setMode(mode, zoneId);
        hvacService.activate(zoneId);
        notifier.send("HVAC activated in " + zoneId, NotificationLevel.INFO);
    }
}
```

<p style={{marginTop: '0.5em', fontSize: '0.85em', color: '#c00'}}>
  To test this, we'd need real sensors, real HVAC, real notifications!
</p>

<aside class="notes">
**What we want to test:**
- If current temp is below target, activate heating
- If current temp is above target, activate cooling
- If within 0.5 degrees, do nothing
- Correct zone is used

**What we DON'T want:**
- Real temperature sensors (need hardware)
- Real HVAC activation (would heat/cool the building!)
- Real notifications (spam real users)

**The solution:** Replace dependencies with test doubles

→ **Transition:** Enter test doubles...
</aside>

## Three Types of Test Doubles

<div style={{display: 'grid', gridTemplateColumns: '1fr 1fr 1fr', gap: '1em', marginTop: '0.5em'}}>
  <div style={{textAlign: 'center'}}>
    <p style={{fontSize: '1.1em', color: '#9370DB'}}><strong>Stubs</strong></p>
    <p style={{fontSize: '0.8em'}}>Return canned answers</p>
  </div>
  <div style={{textAlign: 'center'}}>
    <p style={{fontSize: '1.1em', color: '#9370DB'}}><strong>Fakes</strong></p>
    <p style={{fontSize: '0.8em'}}>Simplified implementations</p>
  </div>
  <div style={{textAlign: 'center'}}>
    <p style={{fontSize: '1.1em', color: '#9370DB'}}><strong>Spies</strong></p>
    <p style={{fontSize: '0.8em'}}>Record what happened</p>
  </div>

<p style={{marginTop: '1.5em', fontSize: '0.9em', color: '#666'}}>
  From simplest to most sophisticated
</p>

<aside class="notes">
**The spectrum:**
- **Stubs:** Dumbest—just return a value
- **Fakes:** Actually work, but simpler than real thing
- **Spies:** Record interactions for verification

**Interview note:** Some testing purists have precise definitions for these terms. The distinctions matter less than understanding when to use each.

</aside>

## Stubs: Return Canned Answers

```java
class StubTemperatureSensor implements TemperatureSensor {
    private final double fixedTemperature;

    public StubTemperatureSensor(double temperature) {
        this.fixedTemperature = temperature;
    }

    @Override
    public double readTemperature(String zoneId) {
        return fixedTemperature;  // Always returns the same value
    }
}
```

<p style={{marginTop: '0.5em', fontSize: '0.85em'}}>
  Ignores the zone, always returns 65.0 — <strong>that's fine!</strong>
</p>

<aside class="notes">
**Stubs are simple:**
- Configure with a value
- Return that value when asked
- Ignore parameters we don't care about

**Why this works:**
- For testing "activate heating when cold"
- We just need the sensor to say "it's cold"
- Don't care about zones, precision, calibration

**Readability (L5):** The stub makes the test's intent clear—we're testing the "cold" case.

→ **Transition:** What about verifying side effects?
</aside>

</Slide>

<Slide>

## Spies: Record What Happened

```java
class SpyHVACService implements HVACService {
    private boolean activateCalled = false;
    private String activatedZone = null;
    private HVACMode setMode = null;

    @Override
    public void activate(String zoneId) {
        this.activateCalled = true;
        this.activatedZone = zoneId;
    }

    @Override
    public void setMode(HVACMode mode, String zoneId) {
        this.setMode = mode;
    }

    // Query methods for tests
    public boolean wasActivateCalled() { return activateCalled; }
    public String getActivatedZone() { return activatedZone; }
    public HVACMode getSetMode() { return setMode; }
}
```

<aside class="notes">
**Spies record:**
- Which methods were called
- With what arguments
- How many times

**Why we need this:**
- adjustToTargetTemperature() returns void
- Can't test by checking return value
- Must verify it called hvacService.activate()

→ **Transition:** Now we can write a complete test...
</aside>

</Slide>

## A Complete Test with Hand-Rolled Doubles

```java
@Test
public void activatesHeatingWhenBelowTarget() {
    StubTemperatureSensor stubSensor = new StubTemperatureSensor(65.0);
    SpyHVACService spyHVAC = new SpyHVACService();
    StubNotificationService stubNotifier = new StubNotificationService();

    ThermostatController controller = new ThermostatController(
        stubSensor, spyHVAC, stubNotifier);

    controller.adjustToTargetTemperature(72.0, "livingRoom");

    assertTrue(spyHVAC.wasActivateCalled());
    assertEquals("livingRoom", spyHVAC.getActivatedZone());
    assertEquals(HVACMode.HEATING, spyHVAC.getSetMode());
}
```

<aside class="notes">
**Trace through the test:**
1. Stub says temperature is 65°F
2. Controller sees target is 72°F (7° delta)
3. Controller should activate heating
4. Spy records that activate() was called

**This test is:**
- Fast (no I/O)
- Focused (tests one behavior)
- Clear (the stubs make the scenario obvious)

**But there's a problem...**

## Mockito: Test Doubles Without the Boilerplate

<p style={{fontSize: '0.95em'}}>
  A <strong>mocking framework</strong> generates test doubles at runtime
</p>

```java
@Test
public void activatesHeatingWhenBelowTarget() {
    // Create test doubles — Mockito generates these at runtime
    TemperatureSensor mockSensor = mock(TemperatureSensor.class);
    HVACService mockHVAC = mock(HVACService.class);
    NotificationService mockNotifier = mock(NotificationService.class);

    // Configure stub behavior
    when(mockSensor.readTemperature("livingRoom")).thenReturn(65.0);

    ThermostatController controller = new ThermostatController(
        mockSensor, mockHVAC, mockNotifier);

    controller.adjustToTargetTemperature(72.0, "livingRoom");

    // Verify spy recordings
    verify(mockHVAC).setMode(HVACMode.HEATING, "livingRoom");
    verify(mockHVAC).activate("livingRoom");
}
```

<aside class="notes">
**Notice the difference:**
- No separate stub/spy classes
- Configure behavior inline with when().thenReturn()
- Verify interactions with verify()

**A Mockito "mock" is really:**
- A stub (when().thenReturn())
- AND a spy (verify())
- Combined into one object

→ **Transition:** Let's break down the key Mockito operations...
</aside>

</Slide>

## Mockito: Create, Configure, Verify

<table style={{margin: '0 auto', fontSize: '0.85em'}}>
  <tbody>
    <tr>
      <td style={{padding: '0.5em 0.8em'}}><code>mock(Class.class)</code></td>
      <td style={{padding: '0.5em 0.8em'}}>Create a test double</td>
    </tr>
    <tr>
      <td style={{padding: '0.5em 0.8em'}}><code>when(...).thenReturn(...)</code></td>
      <td style={{padding: '0.5em 0.8em'}}>Configure stub behavior</td>
    </tr>
    <tr>
      <td style={{padding: '0.5em 0.8em'}}><code>verify(mock).method(...)</code></td>
      <td style={{padding: '0.5em 0.8em'}}>Check spy recordings</td>
    </tr>
    <tr>
      <td style={{padding: '0.5em 0.8em'}}><code>when(...).thenThrow(...)</code></td>
      <td style={{padding: '0.5em 0.8em'}}>Simulate exceptions</td>
    </tr>
  </tbody>
</table>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#666'}}>
  Mockito uses reflection to generate implementations at runtime
</p>

<aside class="notes">
**The four operations you'll use most:**
1. **mock()** — create the double
2. **when().thenReturn()** — "when someone calls this, return that"
3. **verify()** — "assert this method was called"
4. **when().thenThrow()** — simulate failures

**How Mockito works (simplified):**
1. Uses Java reflection to examine the interface
2. Generates a new class implementing all methods
3. Default: methods return null/0/false
4. Records every call (spy)
5. Lets you configure returns (stub)

→ **Transition:** Argument matchers add flexibility...
</aside>

## Argument Matchers: Flexible Matching

```java
// Return 65.0 for ANY zone
when(mockSensor.readTemperature(anyString())).thenReturn(65.0);

// Verify activate was called with any string
verify(mockHVAC).activate(anyString());

// Verify with custom condition
verify(mockNotifier).send(
    argThat(message -> message.contains("HVAC activated")),
    eq(NotificationLevel.INFO)
);
```

<p style={{marginTop: '0.5em', fontSize: '0.85em', color: '#666'}}>
  <code>anyString()</code>, <code>any()</code>, <code>eq()</code>, <code>argThat()</code>
</p>

<aside class="notes">
**Common matchers:**
- anyString(), anyInt(), any() — match any value
- eq(value) — match exactly this value
- argThat(predicate) — match if predicate returns true

**When to use argThat:**
- Don't care about exact message, just that it contains key info
- Testing that a timestamp is "recent" rather than exact

**Rule:** If you use any matcher in a call, ALL arguments must use matchers.

→ **Transition:** What about fakes?
</aside>

</Slide>

## Fakes: When You Need Real Behavior

```java
class FakeUserRepository implements UserRepository {
    private final Map<String, User> users = new HashMap<>();

    @Override
    public void save(User user) {
        users.put(user.getId(), user);
    }

    @Override
    public User findById(String id) {
        return users.get(id);
    }

    @Override
    public List<User> findAll() {
        return new ArrayList<>(users.values());
    }
}
```

<p style={{marginTop: '0.5em', fontSize: '0.85em'}}>
  A working implementation — just simpler than the real database
</p>

<aside class="notes">
**When fakes are useful:**
- Need to save and then retrieve (can't stub that easily)
- Testing multi-step workflows
- Repository patterns (your A4 uses this!)

**Stub vs Fake:**
- Stub: returns canned value, doesn't "work"
- Fake: actually stores/retrieves, but in-memory

**A4 connection:** You'll use fakes for in-memory repositories, mocks for OcrPort.

## The Good: Why Test Doubles Work

<ul style={{fontSize: '0.9em'}}>
  <li><span style={{color: '#060'}}>✓</span> <strong>Speed:</strong> Tests run in milliseconds</li>
  <li><span style={{color: '#060'}}>✓</span> <strong>Determinism:</strong> No flaky tests from real sensors</li>
  <li><span style={{color: '#060'}}>✓</span> <strong>Isolation:</strong> Failures point to the code under test</li>
  <li><span style={{color: '#060'}}>✓</span> <strong>Edge cases:</strong> Easy to simulate errors</li>
</ul>

```java
// Simulating a sensor failure — one line!
when(mockSensor.readTemperature(anyString()))
    .thenThrow(new SensorException("Connection lost"));
```

<aside class="notes">
**Speed:** Thousands of tests in seconds. Run on every save.

**Determinism:** A stub always returns 65.0. The real sensor might return 65.001 or timeout.

**Isolation:** When the test fails, the bug is in ThermostatController, not in network code.

**Edge cases:** How do you test sensor failure with real hardware? Unplug it? With a stub, one line.

**But there are dangers...**

## The Dangerous: Brittle Tests

```java
// This test breaks if you rename activate() to start()
verify(mockHVAC).activate("livingRoom");

// This test breaks if you change the message format
verify(mockNotifier).send(
    "HVAC activated in livingRoom",
    NotificationLevel.INFO
);
```

<p style={{marginTop: '1em', fontSize: '0.95em', color: '#c00'}}>
  Tests coupled to implementation details break on every refactor
</p>

<aside class="notes">
**Brittle test symptoms:**
- Tests break when you refactor, even though behavior is unchanged
- Spend more time fixing tests than writing features
- Developers avoid refactoring because tests will break

**The deeper problem:**
- Are you testing BEHAVIOR or IMPLEMENTATION?
- "It should heat the room" vs "It should call activate()"
- The first is behavior; the second is implementation

**Readability connection (L5):** Tests are documentation. If you're documenting implementation details, you're documenting the wrong thing.

## When to Use Test Doubles

<div style={{display: 'grid', gridTemplateColumns: '1fr 1fr', gap: '1.5em', marginTop: '0.5em'}}>
  <div>
    <p style={{fontSize: '0.95em', color: '#060'}}><strong>Use test doubles when:</strong></p>
    <ul style={{fontSize: '0.8em'}}>
      <li>Dependency is slow or unreliable</li>
      <li>Need to simulate error conditions</li>
      <li>Dependency has side effects (emails, charges)</li>
      <li>Want to verify interactions</li>
    </ul>
  </div>
  <div>
    <p style={{fontSize: '0.95em', color: '#c00'}}><strong>Be cautious when:</strong></p>
    <ul style={{fontSize: '0.8em'}}>
      <li>Mock setup is more complex than code</li>
      <li>Verifying implementation, not behavior</li>
      <li>Mocking types you don't own</li>
    </ul>
  </div>
</div>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#9370DB'}}>
  If mock setup is getting elaborate, consider an integration test instead
</p>

<aside class="notes">
**Good uses:**
- Database, network, hardware
- Simulating timeouts, exceptions
- Verifying notifications were sent
- Avoiding side effects (charges, emails)

**Bad signs:**
- Mock setup longer than the test
- Mocking every little thing
- Mocking third-party libraries (you don't know their contract!)

**The heuristic:** If the mock setup is getting complex, that's a sign to write an integration test instead.

## Connection to Your Assignments

<p style={{fontSize: '1em'}}>
  Your test suites are graded by <strong>mutation testing</strong>
</p>

<ul style={{marginTop: '1em', fontSize: '0.9em'}}>
  <li>We introduce bugs (mutations) into your code</li>
  <li>Your tests should <strong>catch</strong> those bugs</li>
  <li>A test that passes with the bug present = weak test</li>
</ul>

<p style={{marginTop: '1em', fontSize: '0.95em', color: '#9370DB'}}>
  It's not enough for tests to pass — they must detect bugs!
</p>

<aside class="notes">
**Mutation testing explained:**
- We change ">" to ">="
- We delete a line
- We change a constant
- Do your tests catch it?

**Why this matters:**
- You can have 100% coverage with useless tests
- Tests that just call methods without asserting = useless
- Tests that verify the wrong thing = useless

**A4 connection:**
- Test your services with mocks
- But make sure tests actually verify behavior
- Ask: "If this line were wrong, would my test catch it?"

## AI Testing Workflow

<ol style={{fontSize: '0.9em'}}>
  <li><strong>Plan first:</strong> Describe behaviors to test in plain English</li>
  <li><strong>Generate:</strong> Ask AI to create tests from your plan</li>
  <li><strong>Review:</strong> Does it actually test what it claims?</li>
  <li><strong>Break it:</strong> Introduce a bug—does the test fail?</li>
  <li><strong>Validate mocks:</strong> Do configurations reflect reality?</li>
</ol>

<p style={{marginTop: '1em', fontSize: '0.85em', color: '#666'}}>

- "Test that nothing happens when temp is within 0.5 of target"
- "Test that cooling activates when temp is above target"

**Step 4 - Break it:**
- Change ">" to ">=" in your code
- Run the test
- If it still passes, the test is bad

**This is manual mutation testing!**

## Reviewing AI-Generated Tests

```java
// AI generated this — is it good?
@Test
public void testAdjustTemperature() {
    when(mockSensor.readTemperature(any())).thenReturn(70.0);
    controller.adjustToTargetTemperature(75.0, "room1");
    verify(mockHVAC).activate(any());
}
```

<div class="fragment">
<p style={{marginTop: '1em', fontSize: '0.9em', color: '#c00'}}>
  Problems:
</p>
<ul style={{fontSize: '0.8em'}}>
  <li>Doesn't verify heating mode (could be cooling!)</li>
  <li>Uses <code>any()</code> instead of checking zone</li>
  <li>Vague test name doesn't describe expected behavior</li>
</ul>
</div>

<aside class="notes">
**The AI's test passes, but:**
- Doesn't check if it's HEATING vs COOLING
- Doesn't verify the correct zone
- Name is meaningless

**How to fix:**
- Rename: "activatesHeatingWhenBelowTargetByMoreThanHalfDegree"
- Verify: setMode(HVACMode.HEATING, "room1")
- Verify: activate("room1")

**Readability (L5):** Test names are documentation. "testAdjustTemperature" tells you nothing.
